# Benchmark

Reproducible head-to-head of **liteparse-markdown** vs **[pymupdf4llm](https://github.com/pymupdf/RAG)**
on **born-digital PDFs** with ground-truth Markdown. Both are fast, native, no-ML/no-GPU converters.

## Dataset

- **READoc-arXiv** ([`lazyc/READoc`](https://huggingface.co/datasets/lazyc/READoc), **MIT**): real
  born-digital arXiv papers (multi-column, tables, formulas) paired with LaTeX-derived ground-truth
  Markdown. `download.py` fetches a subset: GT `.md` from HuggingFace + the matching PDFs from arXiv
  (by id) — lightweight, avoiding the 1.19 GB zip. The dataset is **not committed** (license + size).
- **Fixtures** (`report`, `prices`, `schedule`, `measurements`) — controlled born-digital docs
  generated by `make_fixtures.py` with exact ground truth (headings, lists, bold/italic and GFM
  tables). Committed, so the harness has self-contained docs and real table (TEDS) signal.

> Why not OmniDocBench? It ships only page **images**; pymupdf4llm reads a PDF text layer and cannot
> process images. Born-digital PDFs are the fair regime for this head-to-head.

## Metrics (`metrics.py`)

Per document and aggregated (**mean and median**, plus a per-subset arXiv/fixtures breakdown):
**text** normalized edit distance / similarity / token-F1 — also a **no-math** variant that strips
LaTeX from both sides since formulas are out of scope for both tools; **heading** and **list**
precision-recall-F1; **table** **TEDS** and **TEDS-S** (structure only). Same families as
OmniDocBench/READoc, computed standalone (rapidfuzz for edit distance, apted for TEDS).

**Timing** is captured per document and, for our library, **split** into `liteparse-java` (parse,
native PDFium) and `liteparse-markdown` (convert) — emitted by the CLI's `--timing` flag in batch mode.

## Run

```bash
cd benchmark
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

python make_fixtures.py          # (optional) regenerate report.pdf
python download.py 40            # fetch ~40 READoc-arXiv docs (GT + PDFs)

# build the library jar first (from repo root): ./gradlew shadowJar
# and have a liteparse-java bundle at ../libs/liteparse-java.jar (or set env vars)

python run.py                    # both tools (batch + timing) -> predictions/ + timing.json
python evaluate.py               # detailed metrics + timing -> results.md / results.csv (and stdout)
```

Jars default to `../build/libs/*-all.jar` and `../libs/liteparse-java.jar`; override with
`LITEPARSE_MD_JAR` / `LITEPARSE_JAVA_BUNDLE`.

## Latest results

See the **Benchmark** section of the [top-level README](../README.md#benchmark) for the summary,
per-document tables, timing and analysis. Headline (44 docs): liteparse-markdown is **~40–100× faster**
and **wins on text (raw and no-math), token-F1, lists and tables on both mean and median**, while being
**competitive (median-tied) on heading detection** (0.500 vs 0.504; mean 0.467 vs 0.508).
